# Gaji 한 페이지 포트폴리오 3안

Date: 2026-05-08

기존 포트폴리오 PDF의 `문제 상황 / 해결 과정 / 결과` 흐름에 맞춰, 개선 보고서 톤으로 재정리했습니다. 최종 추천 순서는 `1안 배포 전 검증 기준 -> 2안 DB 튜닝 -> 3안 검색 방식 선택과 캐시 전략`입니다.

## 1안. AI 답변 검증 기준

### Gaji - AI 답변 배포 전 검증 기준 구성 및 품질 위험 차단

### 문제 상황

AI 답변은 정상 응답처럼 보여도 검색 실패, fallback 은닉, 권한 누락, 내부 지시 노출 문제가 숨어 있을 수 있음

사용자 질문이 원작 장면 질문, What-if 전개, 작품 범위 밖 질문, 출처 요청으로 섞여 있어 하나의 성공 기준으로 판단하기 어려움

실제 Gemini 연결과 SSE 스트리밍에서는 단위 테스트만으로 검색 오판이나 연결 종료 누락을 발견하기 어려움

### 해결 과정

실제 사용자 질문 120개를 익명화한 뒤 장면 탐색형, 인물 해석형, What-if, 소설 밖 질문, 내부 지시 노출 유도 질문으로 분류

검색 검증, 스트리밍 검증, 출처 권한 검증을 분리해 GitHub Actions에서 배포 전 반복 실행할 수 있도록 구성

관련 문단 포함률, 검색/채팅 지연 시간, 대체 응답 발생 여부, 비소유자 접근 차단, 원문 과다 노출, 내부 지시 문구 노출 여부를 검증 항목으로 설정

답변마다 사용 모델, 검색 방식, 대체 응답 사유, 인용 문단, 단계별 소요 시간을 저장해 검색 문제와 생성 문제를 분리 추적

### 결과

검색·스트리밍·출처 권한을 분리한 배포 전 검증 절차를 GitHub Actions에서 반복 실행 가능하게 구축

잠재 오류 1: 작품 범위 밖 질문이 일부 단어 겹침만으로 검색 성공 처리되는 문제 발견

잠재 오류 2: 사용자가 SSE 연결을 끊어도 외부 AI 호출이 계속될 수 있는 문제 발견

두 케이스를 회귀 테스트와 SSE 연결 종료 처리로 보완한 뒤 재검증 통과

## 2안. 대화 내역 검색과 DB 조회 최적화

### Gaji - 대화 내용까지 함께 찾는 종합검색 DB 조회 최적화

### 문제 상황

기존 검색은 대화 제목 중심이라, 긴 토론에서 예전에 나눈 해석이나 인용 메시지를 다시 찾기 어려움

대화 본문과 최근 메시지까지 함께 보여주려면 메시지 테이블을 추가로 읽어야 해서, 데이터가 늘수록 조회량이 급격히 커질 수 있음

실제 개발 데이터 741건만으로는 병목이 드러나지 않아, 운영 규모를 가정한 1,025,000개 메시지 벤치마크가 필요했음

### 해결 과정

제목이나 본문이 검색어와 맞을 가능성이 있는 대화를 먼저 추리고, 그 후보에 대해서만 최근 메시지를 붙이도록 쿼리 재구성

종합검색 응답에 본문 일치 개수, 일치한 문장 일부, 최근 메시지 3개를 함께 포함하고, DB 조회 단계에서 내 대화만 검색되도록 제한

메시지 본문 부분 검색용 인덱스와 대화별 최신 메시지를 바로 읽는 인덱스를 추가

새 인덱스가 기존 대화 ID 단일 인덱스 역할까지 대신할 수 있어 중복 인덱스 제거

### 결과

EXPLAIN ANALYZE 기준 종합검색 전체 실행 시간이 972.709ms에서 40.582ms로 감소

본문 검색은 사용자의 긴 대화 500,000건을 먼저 읽는 방식에서, 본문 후보 6,300건을 먼저 만든 뒤 내 대화와 맞추는 방식으로 변경

최근 메시지 조회는 대화마다 10,000건을 다시 읽어 정렬하던 방식에서, 대화마다 필요한 3건만 최신순으로 읽는 방식으로 변경

## 3안. 검색 방식 선택과 캐시 전략

### Gaji - 질문별 검색 선택과 반복 질문 캐시 최적화

### 문제 상황

사용자 질문은 원문 문장을 그대로 쓰지 않아, 키워드 검색만으로는 관련 문단을 놓치는 경우가 생김

의미 기반 검색을 쓰려면 사용자 질문도 검색용 벡터로 바꿔야 하는데, 모든 질문을 매번 벡터화하면 외부 API 호출과 검증 시간이 늘어남

같은 익명화 사용자 질문을 반복 검증에 다시 쓰면서도, 모델·차원·검색 용도가 바뀌면 잘못된 벡터를 재사용하지 않아야 했음

### 해결 과정

의미 기반 검색, 키워드 검색, 두 결과를 함께 쓰는 방식을 같은 익명화 사용자 질문 묶음으로 비교

표현이 달라진 질문은 의미 기반 검색을 우선 사용하고, 인물명·장면 단어처럼 정확한 단어가 중요한 질문은 키워드 검색이 보완하도록 구성

반복 사용되는 사용자 질문은 검색용 벡터를 캐시에 저장해 같은 검증에서 다시 만들지 않도록 변경

PostgreSQL은 사용자·권한·대화 데이터에 집중시키고, 소설 문단과 벡터는 검색 실험과 재색인 영향이 작도록 별도 저장소로 분리

출처 원문 조회 캐시는 권한 확인 이후 짧은 시간만 재사용하고, 오래된 항목부터 비우도록 상한 관리

### 결과

키워드 검색만으로 놓치는 표현 변형 질문을 확인하고, 의미 기반 검색은 기본 경로로, 키워드 검색은 인물명·장면 단어 보완 경로로 역할을 분리

What-if 창작형 질문은 단일 정답 문단 기준에서 제외하고 원작 맥락 유지 여부로 별도 평가

혼합 검색은 배포 전 검증에서 허용 지연 기준 안에 머무는지 확인하는 보조 지표로만 사용

검색용 질문 벡터 캐시는 모델·차원·검색 용도가 같은 경우에만 재사용되도록 설계했고, 캐시 재실행에서 Gemini Embedding API 재호출 0회 확인
